# 1) 목표 / 비목표

## Goals (MVP 목표)

### 1.1 핵심 기능 목표

#### 1.1.1 Notion 콘텐츠 표시
- Notion 임베드 콘텐츠를 웹 페이지에서 보여준다.
- Notion 공개 공유 링크를 입력받아 iframe으로 임베드한다.
- 콘텐츠가 정상적으로 로드되는지 프리뷰 기능을 제공한다.

#### 1.1.2 블라인드(가림) 기능
- 콘텐츠의 일부를 블라인드(가림/블러/블랙처리) 한다.
- 운영자가 블라인드할 구간을 설정할 수 있다.
- 다양한 블라인드 방식을 지원한다 (섹션 블러, 프리뷰+블러, 키워드 블랙아웃 등).

#### 1.1.3 리드 폼 제출 및 해제
- 리드 폼 제출 후 블라인드를 해제한다.
- 폼 제출 성공 시 즉시 블라인드가 해제되어 콘텐츠를 볼 수 있게 한다.
- 제출 실패 시 적절한 에러 메시지를 표시한다.

#### 1.1.4 상태 유지
- 재방문 시 다시 묻지 않도록 세션/로컬스토리지/쿠키로 해제 상태를 유지한다.
- 기본적으로 30일간 유지하며, 운영자가 기간을 설정할 수 있다 (세션만/7일/30일).

### 1.2 운영자 기능 목표

#### 1.2.1 UX 패턴 선택
- 운영자는 "어떤 UX 패턴으로 게이트할지"를 옵션으로 선택할 수 있다.
- 다음 패턴들을 지원한다:
  - 상단 폼 (Top Form)
  - 하단 폼 (Bottom Form)
  - Entry 모달 (접속 즉시 모달)
  - Floating CTA (우하단 CTA 버튼)
  - Inline Blur Gate (본문 중간 인라인 블러 게이트)
- 복수 선택 가능하며, 충돌 규칙에 따라 조합이 제한될 수 있다.

#### 1.2.2 폼 위치 및 카피 설정
- 운영자는 상단/하단 폼을 각각 켜거나 끌 수 있다.
- 각 폼 위치별로 다른 타이틀/서브타이틀을 설정할 수 있다.
- 상단 폼과 하단 폼은 서로 독립적으로 설정 가능하다.

#### 1.2.3 반응형 설정
- 모바일에서 동일한 경험을 제공한다.
- PC와 모바일 각각 설정을 다르게 할 수 있다 (예: 모바일은 Floating CTA만, PC는 상단+하단 폼).

### 1.3 미리보기 기능 목표

#### 1.3.1 실시간 미리보기
- 설정에 따른 실시간 미리보기 화면을 제공해야 한다.
- 미리보기 화면에는 다음 기능이 포함되어야 한다:
  - 새로고침 버튼: 설정 변경 사항을 반영하기 위해 새로고침할 수 있다.
  - 모바일/PC 전환 토글: 모바일 뷰와 PC 뷰를 전환하며 확인할 수 있다.
- 레이아웃: 좌측에 미리보기, 우측에 설정 패널을 배치한다.

### 1.4 공유 기능 목표

#### 1.4.1 공유 링크 생성
- 공유하기 버튼을 누르면 제공하는 도메인으로 key를 공유한다.
- 하위 경로는 수정 후 저장할 수 있다.
- 중복을 피하기 위해서 중복 검증도 포함한다.

#### 1.4.2 커스텀 도메인
- CNAME 도메인 변경은 추후 제공으로 안내한다.
- MVP에서는 기본 도메인만 제공하며, 커스텀 도메인 UI는 나중에 추가한다.

### 1.5 브랜딩 목표

#### 1.5.1 서비스 로고 버튼
- 추후 유료화도 고려해서 사용자들이 링크에 접속했을 때 우측 하단에 북마크 디자인으로 마진 없이 붙어있는 형태로, 항시 서비스 로고 버튼이 떠 있어야 한다.
- 이 버튼을 통해 서비스로 이동하거나 정보를 확인할 수 있다.

### 1.6 인증 및 사용자 관리 목표

#### 1.6.1 비회원 우선 사용
- 전반적으로 화이트 테마, 비회원으로 선 사용한다.
- 비회원 상태에서도 프로젝트를 생성하고 설정할 수 있다.
- 상단 상시 배너로 로그인 하라고 안내한다.

#### 1.6.2 로그인 유도
- 저장하기/공유하기 할 때 로그인 하라고 모달창으로 안내한다.
- 비회원 상태에서는 임시로 사용할 수 있으나, 저장/공유 시에는 로그인이 필요하다.

#### 1.6.3 이메일 로그인
- 이메일로 로그인을 진행한다.
- 인증은 우선순위가 아니므로 생략한다.
- 가입은 이메일과 패스워드만으로 단순 가입하도록 한다.
- 이메일 인증 단계는 MVP에서 제외하며, 추후 추가할 수 있는 구조로 설계한다.

## Non-goals (MVP에서 하지 않음)

### 2.1 플랫폼 제약

#### 2.1.1 Notion 외 문서 지원
- Notion 외 다른 문서/웹페이지 임베드 지원은 제외한다.
- MVP에서는 Notion 공개 공유 링크만 지원한다.
- 추후 확장 가능성을 고려하여 아키텍처는 확장 가능하게 설계한다.

### 2.2 폼 기능 제약

#### 2.2.1 복잡한 폼 로직
- 고급 분기 로직 (조건부 필드 표시 등)
- 다단계 설문 (Multi-step form)
- 결제 기능 통합
- 파일 다운로드 판매 기능
- MVP에서는 단순한 단일 폼 제출만 지원한다.

### 2.3 권한 관리 제약

#### 2.3.1 고급 권한 체계
- 팀/워크스페이스 기능
- SSO (Single Sign-On) 통합
- 역할 기반 접근 제어 (RBAC)
- MVP에서는 개인 계정 기반의 단순한 소유권 관리만 제공한다.

### 2.4 데이터 통합 제약

#### 2.4.1 CRM 풀동기화
- CRM 시스템과의 실시간 양방향 동기화
- 복잡한 데이터 매핑 및 변환
- MVP에서는 Webhook/CSV Export만 제공한다.
- 사용자가 직접 데이터를 내보내어 CRM에 수동으로 연동할 수 있다.

### 2.5 추가 기능 제약

#### 2.5.1 고급 분석 기능
- 실시간 대시보드
- 복잡한 커스텀 리포트
- A/B 테스팅 기능
- MVP에서는 기본적인 리드 목록 및 CSV 내보내기만 제공한다.

#### 2.5.2 고급 커스터마이징
- 완전한 테마 커스터마이징
- 커스텀 도메인 (CNAME) 실제 연결 기능
- 커스텀 폼 스타일링 (MVP에서는 기본 스타일만 제공)
- MVP에서는 기본 화이트 테마와 제한된 커스터마이징만 제공한다.

#### 2.5.3 다국어 지원
- 다국어 UI/메시지
- 로케일별 폼 필드
- MVP에서는 한국어와 영어만 지원 (필요시).

#### 2.5.4 모바일 앱
- 네이티브 모바일 앱
- MVP에서는 웹 앱만 제공하며, 모바일 브라우저 최적화는 포함한다.

## MVP 범위 결정 원칙

### 3.1 핵심 가치 제공 우선
- "Notion 콘텐츠를 블라인드하고 리드를 수집한다"는 핵심 가치를 최우선으로 구현한다.
- 부가 기능은 핵심 가치가 검증된 후 추가한다.

### 3.2 단순성 유지
- 복잡한 설정보다는 간단하고 직관적인 인터페이스를 우선한다.
- 고급 기능은 "고급설정" 섹션으로 숨겨서 기본 사용자는 단순하게 사용할 수 있게 한다.

### 3.3 확장 가능한 구조
- MVP에서 제외하는 기능이라도 추후 추가할 수 있도록 아키텍처를 설계한다.
- 데이터 모델과 API 설계 시 확장성을 고려한다.

### 4.4 사용자 피드백 기반 진화
- MVP 출시 후 사용자 피드백을 수집하여 우선순위를 재조정한다.
- Non-goals에 포함된 기능이라도 사용자 요구가 크면 우선순위를 상향 조정할 수 있다.
